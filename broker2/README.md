# broker

Lean in-memory pub/sub broker. A leaner, faster alternative to [`broker1`](../broker1/README.md) (the previous implementation) for callers that don't need its richer feature set (async publish, observability hooks, stats, queue groups, panic recovery, per-message UUID/timestamp).

---

## Status

Experimental — internal package, not yet wired into any module. Topic matching supports exact strings and a single trailing/inline `*` wildcard segment (via `internal/ds.Trie`), not the full `matcher` pattern grammar (`>`, multi-segment complex patterns, etc.) that `broker1` supports.

---

## Quick start

```go
import "github.com/dangduoc08/ginject/broker"

b := broker.New()

id, _ := b.Subscribe("user.created", func(m *broker.Message) {
    fmt.Printf("new user: %v\n", m.Payload)
})

b.Publish("user.created", map[string]string{"name": "alice"})

b.Unsubscribe("user.created", id)
```

## Broker

```go
type Broker interface {
    Subscribe(topic string, handler MessageHandler) (uint64, error)
    Publish(topic string, payload any) error
    PublishAsync(topic string, payload any) error
    Unsubscribe(topic string, id uint64) error
    Subscriptions() map[string][]uint64
}

func New() Broker // returns a *MemoryBroker
```

- `Subscribe` returns `ErrEmptyTopic` for `""` and `ErrNilHandler` for a nil handler. On success it returns a non-zero `id` unique within that broker instance. Multiple handlers may be registered on the same topic; each gets its own `id`.
- `Publish` returns `ErrEmptyTopic` for `""`. Publishing to a topic with no subscribers is a no-op (`nil` error). Handlers run synchronously, in registration order, on the caller's goroutine. If a handler panics, the panic is recovered and silently dropped — it does not stop sibling handlers from running and does not propagate to the caller. There is no hook to observe a recovered panic (unlike `broker1`'s `OnPanic`).
- `PublishAsync` validates `topic` synchronously (same `ErrEmptyTopic` as `Publish`), then dispatches in a new goroutine — `go b.Publish(topic, payload)` — and returns `nil` immediately without waiting for handlers to run. There's no worker pool, no queue, no backpressure, and no way to observe completion or failure: each call spawns its own goroutine, unbounded. Handler panics are still recovered (it reuses `Publish` internally), so it cannot crash the spawned goroutine.
- `Unsubscribe(topic, id)` returns `ErrEmptyTopic` for `""`. It is idempotent — calling it twice with the same `(topic, id)`, or with an `id` that doesn't exist on that topic, is a safe no-op (`nil` error). It removes only the handler tied to that exact `(topic, id)` pair, leaving other handlers on the same topic (and all other topics) untouched. When that was the topic's last handler, the topic itself is removed from the index.
- `Subscriptions()` returns a snapshot `map[topic][]id` of every currently-registered subscription. The returned map and slices are independent copies — later `Subscribe`/`Unsubscribe` calls do not mutate a snapshot already returned.

```go
var (
    ErrEmptyTopic = errors.New("broker: topic must not be empty")
    ErrNilHandler = errors.New("broker: handler must not be nil")
)
```

## Wildcard matching

```go
b.Subscribe("user.*", handler) // matches "user.created", "user.deleted", ...
b.Subscribe("*", handler)      // matches any topic
```

An exact-topic match always takes precedence over a wildcard match for the same `Publish` call — `find` resolves the literal topic via a hash lookup before falling back to the wildcard trie.

---

## Concurrency model

- `Subscription` (the internal trie+hash index) is guarded by a single `sync.RWMutex`. `insert`/`remove`/`list` take `Lock()`/`RLock()` as appropriate; `find` takes `RLock()`. There is only one lock in this type, so no lock-ordering concerns.
- Subscription ids are generated by an `atomic.Uint64` counter, independent of the mutex — only uniqueness is required, not ordering relative to map state.
- `Publish` never holds the lock while invoking handlers: `find` clones the handler slice under `RLock`, releases it, then dispatches — safe even if a handler calls back into `Subscribe`/`Publish`/`Unsubscribe`.
- `Subscribe`, `Publish`, `Unsubscribe`, and `Subscriptions` run synchronously on the caller's goroutine; only `PublishAsync` spawns a goroutine (one per call, unmanaged — see above).

## Differences from `broker1`

| | `broker1` | `broker` |
|---|---|---|
| Message | `ID` (UUID), `Topic`, `Payload`, `Timestamp`, `Metadata` | `Topic`, `Payload` |
| Pattern grammar | exact, `*`/`>` global, `user.*`, `user.>`, complex multi-wildcard | exact, single wildcard segment |
| Fan-out across pattern kinds | yes (exact + prefix + global + complex all fire together) | no — `find` resolves to one match: exact, else single wildcard fallback |
| Async publish | `PublishAsync` with bounded worker pool + queue (`ErrAsyncQueueFull`/`ErrNoAsyncWorkers`) | `PublishAsync` spawns one unbounded goroutine per call, no queue, no backpressure |
| Queue groups | `SubscribeQueue` (competing consumers) | not implemented |
| Once | `Once` (fires at most once) | not implemented |
| Stats / hooks | yes | not implemented |
| Panic recovery | configurable (`RecoverPanics` + `OnPanic` hook) | always on, unconditional, no hook — a panicking handler is silently recovered |
| Publish cost | UUID generation + stats counters + hook checks per call | hash lookup + slice clone, no per-call allocation beyond the result |

Use `broker` when raw publish throughput matters more than the above features; use `broker1` otherwise.
